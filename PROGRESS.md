## 2026-07-17 — MP input buffer fix (branch `mp-inputbuffer-fix` — the sprint-stop hard rubberband, the last residual movement artifact)

- **Root cause (confirmed by the deterministic repro): the server applied the client's LATEST-held `MoveInput` once per tick (`TryGetHeldInput`) instead of consuming the input stream in seq order.** Under jitter/reorder/loss on the input channel the number of distinct input-ticks the server integrates drifts from the number the client predicted: two inputs landing in one server-tick window integrate only the newest (a client-predicted tick of motion is silently skipped), a gap tick re-integrates a stale input. During steady motion the drift is invisible (the same held axes integrate either way) — it materializes at input CHANGES. At walk the per-tick delta is ~9 cm (the dead-zone hides most of it); at SPRINT one tick is 0.14 m, so a 1-3 tick count mismatch is a 0.15-0.4 m gap that resolves as one hard correction the instant the player stops. Plus a second, subtler artifact the repro exposed: held-model write-backs repeat a stale seq while coast ticks keep advancing the position, and the 25 Hz snapshot sampler can pair a coast-advanced position with a seq the client only predicted once — a phantom ±1-tick correction that immediately reverses (the constant jitter wobble while sprinting).
- [x] **The fix mirrors real Unturned** (`PlayerInput.cs` — `serversidePackets` queue `:1054`, enqueue with out-of-order reject `:1487`, `Peek`/`Dequeue` one per sim frame `:1723-1734`): a per-peer in-order jitter buffer in `core/UnturnedNet/PlayerReplication.cs` (`TryConsumeInput` + queue state on `PlayerEntity`), consumed by `game/PlayerNetSync.cs` instead of the held view.
