# PLAN — Combat for the PEI Multiplayer Client

*A joined player can shoot and the shots matter: kill zombies with real hit feedback, and damage/kill other players with server-authoritative vitals, death, and respawn synced to every client. The C1–C6 milestone made the client world real; this closes the pacifist gap. Test-server scope: simple > complete, SP stays byte-identical.*

**The single highest-leverage fact:** the hard half is DONE and tested. `ServerCombat` validates, steps bullets against server positions with head/torso/leg zones, resolves melee on a deferred timer, flies grenades, and owns death/respawn/kill-credit — proven end-to-end over the real wire by the L0 battery (`ServerCombatTests.cs`, 8 tests incl. two-client kill+respawn parity at `:157-221`). The commands are registered (`NetWorldHost.cs:82-89`) and the client send methods exist (`NetWorldHost.cs:434-464`). What's missing is *wiring*: the shell fires with local SP authority instead of calling `SendFire`, the seven render events have zero subscribers in `game/`, and the C2 invulnerability guards make every avatar body a ghost. This plan is mostly composition, with exactly two real new mechanisms (the fx-only local fire path; the replica-driven shell vitals).

## 1. Current state — verified map

### 1.1 Server combat: built, validated, replicated, L0-tested

- **Fire**: `ServerCombat.OnFire` (`ServerCombat.cs:171-199`) rejects dead/missing senders (`:173`), mid-reload (`:176`), rate abuse — min gap Firerate+1 ticks, the SP off-by-one rule (`:177`), empty mag (`:178`), fake muzzle origins beyond 3 m of the avatar (`:179`, `MaxAimOriginOffset` `:40`), and degenerate aim vectors (`:180-182`). Accepted shots spawn server bullets stepped through the exact SP `BallisticsMath` model (`:263`, IEEE-identical to the shell's local step, `PlayerController.cs:2009-2012`).
- **Hit resolution** against SERVER positions: players via segment-vs-cylinder with the head(≥1.45 m, 3.0×)/torso(≥0.78 m, 1.0×)/leg(0.6×) zones (`:272-277`, `:302`, constants `:90-96`); zombies flat-damage like SP `StepBullets` (`:313`); world geometry via the host-supplied `WorldRay` (`:287-292`) — the dedicated server wires real Godot raycasts (`DedicatedServer.cs:58`, `:127-138`).
- **Melee** = server-side deferred-hit timer re-evaluating targets at land time (`:209-216`, `:344-404`); **grenades** = server entities with real gravity/bounce, snapping through `ProjectileReplication`, exploding with squared falloff vs players / linear vs zombies (`:218-230`, `:406-461`).
- **Death/respawn**: `ApplyPlayerDamage` (`:470-488`, **private**) decrements `HealthExact`, flips `Alive=false` at 0, bumps `Deaths`, sets `RespawnAtTick = tick + 175` (3.5 s, `:96`), drops the corpse's held input at the choke point (`:484` + the `IsAlive` validate on MoveInput, `NetWorldHost.cs:81`), credits the killer (`:485`), broadcasts `PlayerDied` (`:486-487`). `Respawn` (`:246-256`) restores 100 HP, `ServerTeleport`s the entity to `SpawnPos` (captured at join, `NetWorldHost.cs:95`) and broadcasts `PlayerRespawned`.
- **Where the damage lives — the recon question**: `ServerCombat` damages the **replication entity** (`PlayerCombatReplication.CombatEntity.HealthExact/Health/Alive`), *never* the Godot `PlayerController`. The NetAvatar body on the dedicated server is movement-only; on death its input dries up (choke point) and on respawn `PlayerNetSync` adopts the teleport through its snap-to-entity path (explicitly listed: "combat respawn", `PlayerNetSync.cs:19-21`, `:90-98`). So "turning death on" is a client-rendering + damage-routing problem, not a server-HP problem.
- **Replication**: `CombatEntity` puts `Alive` + coarse `Health` byte + `Kills`/`Deaths` on the wire (`CombatReplication.cs:466-473`); `HealthExact/Ammo/reload/respawn ticks/SpawnPos` are server-only (`:355-362`). `SystemPlayerCombat` (id 2) is **in the `EnableSyncCheck` set** (`NetWorldHost.cs:137`) and its `StateHash` covers alive/health/kills/deaths (`CombatReplication.cs:451-464`) — any client-side write to the replica trips the desync banner.
- **Proven by L0, not L1**: the combat battery is `tests/UnturnedNet.Tests/ServerCombatTests.cs` (MemNetwork + real `NetWorldServer`/`NetWorldClient`, full wire): validation gates (`:86-155`), **two-client kill → both replicas agree on death/credit by StateHash → server respawn at spawn with 100 HP, seen by both** (`:157-221`), headshot one-shot (`:224-246`), zombie kill credit + events (`:251+`), melee deferred-hit (`:296+`), grenade flight/falloff (`:325+`). There are **no** `net.*combat*` L1 tests today (`NetTests.cs` name sweep confirms) — the in-engine lane starts with this plan.

### 1.2 The client shell: a full SP fire path, none of it on the wire

`ClientWorldSession.SpawnShell` builds a normal `PlayerController` (NOT NetAvatar) and calls `shell.EquipUnarmed()` with the comment "combat over the wire is deferred, §6" (`ClientWorldSession.cs:171-177`). But the shell is a full SP client: it gets the demo inventory **including real guns** (`PlayerController.cs:1616-1618`), so a joined player can equip an Eaglefire today and:

- **LMB** → `StartFire` → `Fire()` (`PlayerController.cs:1667-1677`, auto-poll `:2560`): full local authority — ammo decrement, recoil/shake, muzzle light, tracer, casings — and a local simulated bullet (`:1923-1985`) stepped by the local `StepBullets` (`:2002-2040`).
- Local bullets **collide with zombie puppets** (puppet capsules keep the enemy layer, `ZombieController.cs:82`) and show a **hitmarker** (`PlayerController.cs:2021`) — but `ApplyDamage` no-ops on puppets (`ZombieController.cs:200`), so the hitmarker lies: fx fire, nothing dies.
- **G** melee (`:1728-1729`) → `MeleeAttack`/`ApplyMeleeHit` iterates the `"zombies"` group (`:921`) — puppets never join it (`ZombieController.cs:81`), so MP melee whiffs entirely.
- **H** grenade (`:1730-1731`) → `ThrowGrenade` spawns a REAL local `Grenade` (`:1012-1021`) whose blast calls `Thrower.Explode` (`Grenade.cs:41`) — which can locally damage **the shell itself** (`PlayerController.cs:977`). A joined client can locally kill themselves with their own grenade while the server disagrees.
- **The shell's local vitals are fully live** (it is not a NetAvatar): `UpdateVitals` steps hunger/thirst/regen (`:1498-1511`), `TakeDamage` works (`:1417-1445`), local `Die()` plays the death cam + corpse (`:1447-1473`) and the local `_deathTimer` respawns at the LOCAL `Spawn` (`:2518-2524`, `:1475-1492`) — after which the reconciler yanks the node back toward the server entity. Fall damage (`:2613`, `:941-947`) and the OOB kill (`:2512`) are similarly local-only.
- HUD health is a live lambda over `Player.Health` (`HUD.cs:65` → `_vitals.Health`, `PlayerController.cs:42`) — syncing server HP to the HUD is one assignment.

### 1.3 The unconsumed render hooks

`NetWorldClient` declares and fires all seven combat facts (`NetWorldHost.cs:285-292`, registered `:348-355`): `HitConfirmed` (shooter-only, `ServerCombat.cs:503-507`), `ImpactFx` (broadcast per bullet end, `:509-513`), `PlayerDied`, `PlayerRespawned`, `ZombieHit`, `ZombieDied`, `GrenadeExploded`. **A grep of `game/*.cs` finds zero subscribers** — the session consumes only `VehicleEntered/Exited` (`ClientWorldSession.cs:86-87`) and the Phase 6/8 facts (via the views). Likewise `Client.Projectiles` (in-flight grenades) has **no view** — a server grenade is invisible until it explodes. Zombie deaths, in contrast, already render for free: `ServerCombat` sets the replicated anim byte to `Dead` (`ServerCombat.cs:319`) and `ZombiePuppets` plays the ragdoll off it (`ZombiePuppets.cs:10`).

### 1.4 The invulnerability guards

- `PlayerController.TakeDamage:1419` — `if (NetAvatar) return;` ("C2 v1: server avatars are invulnerable to LOCAL damage… an unreplicated death would desync every client"). This is what eats zombie swings on the dedicated server (`ZombieController.cs:370` calls it directly).
- `CheckFallDamage:943` — `if (NetAvatar) return;` (also keeps the broken-legs flag from eating the wire's jump bit).
- `UpdateVitals:1500` — `if (NetAvatar) return;` (no starvation death on a server body).
- The OOB kill is gated `!NetAvatar` too (`:2512`).

So today: server bodies can't be hurt by anything; the client shell can only be hurt by itself (locally, unreplicated). Server-side player damage (`ApplyPlayerDamage`) is reachable **only** through the wire commands nobody sends.

## 2. Target

1. The shell's trigger pulls route over the wire: `Fire()`/`MeleeAttack()`/`ThrowGrenade()`/`StartReload()` call `Client.SendFire/SendMelee/SendGrenade/SendReload` (`NetWorldHost.cs:434-464`) through null-default delegates; the local bullet/swing keeps ALL its fx (tracer, recoil, muzzle, casings, swing anim) but applies **zero damage** — MP damage is a server fact.
2. The client renders the server's facts on the real world: hitmarkers from `HitConfirmed`, impact decals/blood from `ImpactFx`, puppet blood + kill feed from `ZombieHit`/`ZombieDied`, blast fx from `GrenadeExploded`, corpses + death cam + respawn from `PlayerDied`/`PlayerRespawned`, in-flight grenades from a `Projectiles` view.
3. Vitals become server-authoritative for every networked player: the NetAvatar guard is replaced by a **route** (damage forwards to the combat entity instead of vanishing), the shell's HUD HP mirrors the `CombatState` replica, the shell's death/respawn is driven by the server events, and zombies are finally dangerous in MP.
4. SP byte-identical throughout: the direct `StepBullets`/`TakeDamage`/`Die` path is untouched when the delegates/flags are unset (their SP default).

## 3. Phased breakdown

Two phases. D1 is the low-risk half (new code is almost all client-side fx wiring; the one server change is a safety toggle); D2 flips authority over the shell's life, which touches the vitals/death/respawn state machine and the reconciler. Each lands separately, `./test.sh` + `--visual` green, zero re-baselined goldens.

## 4. Server-auth vitals design — the crux, condensed

- **Who owns HP**: the server's `CombatEntity.HealthExact` (float, server-only) with the coarse `Health` byte + `Alive` on the wire (`CombatReplication.cs:349-356`). It ALREADY replicates to every client and is hash-checked. There is no second health pool to reconcile: the Godot bodies (server avatars) and the shell never own MP health — the shell **displays** the replica, the avatar **forwards** damage into the entity.
- **The guards flip by becoming routes**: `TakeDamage:1419` / `CheckFallDamage:943` / the OOB gate `:2512` route into `NetDamageSink` → `DamagePlayerExternal` → the one shipped `ApplyPlayerDamage` lifecycle. `UpdateVitals:1500` stays a guard (no server hunger sim — starvation simply can't kill in MP v1, consistent with test-server scope).
- **Death stops the shell** through the pieces that already exist: server-side the corpse's input is dropped at the dispatch choke point (`NetWorldHost.cs:81`) and cleared (`ServerCombat.cs:484`); client-side `NetDie()` reuses the SP death cam/corpse and `ShellStep` goes quiet. **Respawn re-seats it** off the server's `ServerTeleport` (`ServerCombat.cs:253`) — the shell node adopts the entity position on the `PlayerRespawned` event, exactly the adoption pattern `PlayerNetSync` (`:90-98`) and the C6 exit (`ClientWorldSession.cs:194-213`) already use.
- **Why it hash-converges**: all four mutation sources (wire combat, zombie sink, fall/OOB sink, respawn clock) run inside `TickSimulation` on the server (`NetWorldHost.cs:156-170` — `Combat.Step` before replication, `PlayerNetSync` between sim and publish); clients only mirror. The desync detector (1 Hz, SystemId 2 in the set) is the regression alarm for anyone later "fixing" a visual by writing a replica.

1. **The `Fire()` fx-vs-damage split** — the method is dense (rechamber/burst/shotgun/rocket branches, `:1923-1985`) and `StepBullets` mixes fx and damage per collider type (`:2016-2035`).
